# Plan — `phax init`, a local JSON Schema, and `phax schema upgrade`

Implements `docs/specs/10-init-command.md`.

## Design decisions (resolved during planning)

The spec leaves the `$schema` reference strategy open (§7–§12). After inspecting
the codebase and distribution model, the strategy below is **chosen**. If you
disagree, stop before phase-01 — it shapes every phase.

### Runtime validation is the authority; the editor schema is a convenience

`loadConfig` already decodes every config through `decodePhaxConfig`
(`onExcessProperty: "error"`) and returns a structured `ConfigValidationError`
with field-level detail from `formatParseError`. Every config-consuming command
already calls `loadConfig`. **The installed binary's structural validation is the
source of truth** — a config is acceptable iff the running binary's
`PhaxConfigSchema` accepts it.

This reframes the `$schema` question entirely: the editor schema is a
best-effort convenience, fully backstopped by the binary. If the editor schema
is stale or ahead of the binary and shows a false green, the binary catches it
immediately at `run`/`resume` with the truthful, blocking error. So the editor
schema does **not** need to be perfectly in sync, and we do not host it remotely
or pin it to versions.

### Schema reference: a local file generated by the binary

`phax init` generates the JSON Schema from the live `PhaxConfigSchema`
(via `JSONSchema.make`, already used in `src/schemas/phaxPlan.ts`) and writes it
into the project as `phax.schema.json`, referenced by a relative
`"$schema": "./phax.schema.json"`.

Why this over the alternatives we evaluated:

- **Remote URL (CLI-version-pinned, config-version-keyed, or floating)**: any
  independently-sourced schema can skew from the installed binary, because the
  binary version the user runs is independent of any URL the config commits to.
  Floating skews *ahead* of the binary (false green — dangerous); pinning skews
  *behind* on upgrade (false red — stale). Hosting also adds infra and an offline
  gap. Since runtime validation is the authority, none of this is worth it.
- **`node_modules`-relative reference**: only resolves for a local npm
  devDependency; phax's npm package is a thin launcher that downloads a binary,
  so this is fragile and breaks for global/binary installs.

A locally generated file always matches the binary that wrote it, works in every
install mode, needs no network, and any staleness after a CLI upgrade is only
ever the safe direction (editor behind binary) — refreshed explicitly by
`phax schema upgrade`.

This also resolves spec AC 11 ("the release process ensures the schema is present
wherever the chosen strategy requires it"): because the schema is generated at
runtime by the binary, there is **nothing to package or publish** — no release
step, npm `files` entry, or hosted asset is required. The release pipeline is
unchanged by this plan.

### Keeping the schema fresh: `phax schema upgrade`

After upgrading the phax binary, the user runs `phax schema upgrade` to
regenerate `phax.schema.json` from the new contract. It touches only the schema
file, never `phax.json`, so it cannot clobber hand-edited config. It is
idempotent and create-if-missing.

### `version` field (decision — reversible)

The `version: Schema.Literal(1)` field is **not** a reliable compatibility proxy
— the structural schema is the real contract, and runtime decode already
accepts/rejects precisely. This plan **keeps the field unchanged** (the
non-breaking choice; removing it would itself break every existing config under
`onExcessProperty: "error"`). Its role is reframed as a future breaking-change
marker, not a validation gate. No phase modifies it. Reverse this decision before
phase-01 if you want it dropped.

### Canonical config file

`phax.json` at the project root. `loadConfig` discovers it by walking up from cwd
to the git root. `phax init` writes `phax.json` and `phax.schema.json` into the
current working directory. No new filename is introduced.

### Minimal generated config

`PhaxConfigSchema` requires `version`, `project`, `state`, and a non-empty
`gateProfiles`, so the minimal-but-valid config is:

```json
{
  "$schema": "./phax.schema.json",
  "version": 1,
  "project": { "name": "<cwd-basename>", "type": "single-package" },
  "state": { "root": "~/.phax" },
  "gateProfiles": {
    "fast": ["echo 'replace with your gate commands in phax.json'"]
  }
}
```

`gateProfiles` cannot be omitted (required, commands must be non-empty), so the
template seeds one `fast` profile with a labelled placeholder. This passes
`decodePhaxConfig`/`loadConfig` (acceptance criteria 3 & 12); the init output
tells the user to replace it. `project.name` is the basename of the working
directory (fallback `"project"` if empty).

### Repo dogfood — already done

The repo's own `phax.json` already references `./phax.schema.json` and a
committed `phax.schema.json` exists (fixed during planning). Once phase-01/02
land, that file becomes reproducible from `getPhaxConfigJsonSchema()`, so no
separate dogfood phase is needed.

## Required commands

- (none)

This plan introduces no new tool, runtime, or CLI. `phax init` and
`phax schema upgrade` use Node's `fs` and Effect Schema only; verification uses
the existing `pnpm` gate commands already in `phax.json`.

## phase-02 — Init project command and schema-file writer {#phase-02-init-app}

**Recommended model:** claude-sonnet-4-6
**Recommended effort:** medium

Add the application-layer logic that produces a minimal valid `phax.json` plus a
local `phax.schema.json`, and the shared schema-file writer reused by
`phax schema upgrade`. This is the core of both commands, independent of the CLI.

### Detailed instructions

- Create `src/app/initProject.ts`, implemented synchronously with Node's `fs`
  (`existsSync`, `writeFileSync`, `readFileSync`), matching the port-free style of
  `src/app/loadConfig.ts` and `src/cli/commands/validate.ts`. Do not introduce
  Effect `FileSystem` layer wiring.
- Export `writeConfigSchemaFile(targetPath: string): { changed: boolean }`:
  - Serialize `getPhaxConfigJsonSchema()` (phase-01) with
    `JSON.stringify(schema, null, 2)` plus a trailing newline.
  - If `targetPath` exists and its current content already equals the serialized
    output, return `{ changed: false }` and do not rewrite.
  - Otherwise write `targetPath` and return `{ changed: true }`.
- Export `initProject(input: { cwd: string; force?: boolean }): InitResult`:
  - `InitResult` is a discriminated union:
    - `{ kind: "already_initialized"; configPath: string }` when
      `<cwd>/phax.json` exists and `force` is not set.
    - `{ kind: "created"; configPath: string; schemaPath: string;
      schemaReference: string }` otherwise. `schemaReference` is
      `"./phax.schema.json"`.
  - Build the minimal config (see "Minimal generated config" above):
    `$schema: "./phax.schema.json"`, `version: 1`,
    `project: { name: basename(cwd) || "project", type: "single-package" }`,
    `state: { root: "~/.phax" }`,
    `gateProfiles: { fast: ["echo 'replace with your gate commands in phax.json'"] }`.
  - Write `<cwd>/phax.json` with `JSON.stringify(config, null, 2)` + newline
    (overwriting only when `force`), then call `writeConfigSchemaFile` on
    `<cwd>/phax.schema.json`.
  - The generated config object must round-trip through `decodePhaxConfig` as a
    `Right` (assert in tests).

### Planned files to create

- `src/app/initProject.ts`
- `tests/unit/initProject.test.ts`

### Planned files to edit

- (none)

### Optional files that may be edited

- (none)

### Boundary contracts

Consumers (phase-03, phase-04 CLI): `initProject` returns the `InitResult` union
and performs all filesystem effects; `writeConfigSchemaFile` returns
`{ changed }` so `phax schema upgrade` can report updated-vs-current. Consumer of
phase-01: imports `getPhaxConfigJsonSchema` from `src/schemas/phaxConfig.ts`.

### Test strategy

Unit tests (application command, real tmp dirs, written before implementation —
stable contract), following the `mkdtempSync` pattern in
`tests/unit/loadConfig.test.ts`:

- `initProject({ cwd })` in an empty dir returns `kind: "created"`, writes both
  files, and the written config parses to a `Right` via `decodePhaxConfig`.
- The written config has `$schema === "./phax.schema.json"` and
  `project.name === basename(cwd)`.
- A second `initProject({ cwd })` (no force) returns `already_initialized` and
  leaves `phax.json` byte-identical (write a sentinel first to prove no
  overwrite).
- `initProject({ cwd, force: true })` overwrites an existing `phax.json`.
- `writeConfigSchemaFile` returns `{ changed: true }` on first write,
  `{ changed: false }` on a no-op second call, and `{ changed: true }` after the
  file is mutated.

### Implementation order

1. Write `tests/unit/initProject.test.ts`.
2. Implement `writeConfigSchemaFile` then `initProject` in
   `src/app/initProject.ts`.

### Excluded scope

- CLI registration and output (phase-03/04).
- Interactive setup, config migration (spec non-goals).
- Editing any existing `phax.json` beyond create / `--force` overwrite.

### Expected handoff content

- The module path `src/app/initProject.ts`, the `initProject` and
  `writeConfigSchemaFile` signatures, and the full `InitResult` union.
- The exact generated `$schema` value and the two filenames.
- Any deviation from the planned file lists, with the reason.

### Verification

- The `full` gate profile configured in `phax.json`.

### Commit subject

feat(init): add initProject command and config-schema writer

### Commit body

Add the application-layer initProject command (minimal valid phax.json plus a
local ./phax.schema.json) and writeConfigSchemaFile, an idempotent writer reused
by phax schema upgrade. initProject refuses to overwrite an existing config
unless forced. Covered by unit tests for creation, the already-initialized
no-overwrite path, force overwrite, and writer idempotency.

## phase-03 — `phax init` CLI command {#phase-03-init-cli}

**Recommended model:** claude-sonnet-4-6
**Recommended effort:** low

Expose `phax init` on the CLI, wiring `initProject` to the output port with clear
diagnostics, and document the command.

### Detailed instructions

- Create `src/cli/commands/init.ts` exporting
  `runInit(opts: { force?: boolean }, out: OutputPort): number`, modeled on
  `src/cli/commands/validate.ts`.
  - Call `initProject({ cwd: process.cwd(), force: opts.force })`.
  - On `already_initialized`: print `PHAX is already initialized: <configPath>`
    plus a hint to use `--force`; return exit code `1` (soft failure for
    scripts).
  - On `created`: print the diagnostics required by spec §11 —
    `Created PHAX config: <configPath>`,
    `Created JSON Schema: <schemaPath>`,
    `Schema: local generated schema (<schemaReference>)`,
    and a next-command line:
    `Next: set your gate commands in phax.json, then run \`phax validate\` or \`phax run\`.`
    Return exit code `0`.
- Register the command in `src/cli/main.ts`: `program.command("init")` with
  `.option("--force", "Overwrite an existing phax.json")` and an action calling
  `runInit` then `process.exit`.
- Update `README.md`: add `phax init` to the command list — creates `phax.json`
  and `phax.schema.json`, refuses to overwrite without `--force`, references a
  local JSON schema.

## phase-04 — `phax schema upgrade` command {#phase-04-schema-upgrade}

**Recommended model:** claude-sonnet-4-6
**Recommended effort:** medium

Add `phax schema upgrade`, which regenerates the project's `phax.schema.json` from
the installed binary's contract. It touches only the schema file, never
`phax.json`, so it cannot clobber hand-edited config. This is how users refresh
the editor schema after upgrading phax.

### Detailed instructions

- The schema lives next to `phax.json`. Expose the existing config locator so the
  command can find the config's directory without decoding it: in
  `src/app/loadConfig.ts`, export a `locatePhaxConfig(cwd: string): string |
  undefined` that returns the path to the nearest `phax.json` (reusing the
  existing private `findGitRoot` + `findPhaxConfig` logic). Do not change
  `loadConfig`'s behavior.
- Add `upgradeConfigSchema(cwd: string): UpgradeResult` to
  `src/app/initProject.ts`:
  - `UpgradeResult` is `{ kind: "no_config" }` |
    `{ kind: "updated"; schemaPath: string }` |
    `{ kind: "current"; schemaPath: string }`.
  - Use `locatePhaxConfig(cwd)`; if no `phax.json` exists, return `no_config`.
  - Otherwise compute `<configDir>/phax.schema.json`, call
    `writeConfigSchemaFile` (phase-02), and map `{ changed }` to `updated` /
    `current`. Note: this works even if the existing `phax.json` is invalid — the
    schema file is independent of config content.
- Create `src/cli/commands/schema.ts` exporting
  `registerSchemaCommand(program, out)` (modeled on
  `src/cli/commands/agent.ts` / `security.ts`) with a `schema` command and an
  `upgrade` subcommand whose action calls a `runSchemaUpgrade(out): number`.
  `runSchemaUpgrade` calls `upgradeConfigSchema(process.cwd())` and maps the
  `UpgradeResult`:
  - `no_config` → print `No phax.json found. Run \`phax init\` first.`; return 1.
  - `updated` → print `Updated <schemaPath>`; return 0.
  - `current` → print `<schemaPath> is already up to date`; return 0.
- Register `registerSchemaCommand` in `src/cli/main.ts` next to the other
  `register*Command` calls.
- Update `README.md`: document `phax schema upgrade` — regenerates the local
  schema from the installed phax version; run it after upgrading phax; never
  modifies `phax.json`.

### Planned files to create

- `src/cli/commands/schema.ts`
- `tests/unit/cli/schemaUpgrade.test.ts`

### Planned files to edit

- `src/app/initProject.ts`
- `src/app/loadConfig.ts`
- `src/cli/main.ts`
- `README.md`

### Optional files that may be edited

- (none)

### Boundary contracts

Surface → application: `runSchemaUpgrade` calls `upgradeConfigSchema`, which uses
`locatePhaxConfig` (new export in `src/app/loadConfig.ts`) and
`writeConfigSchemaFile` (phase-02). Stable need: the CLI must distinguish
no-config / updated / already-current to render the right line and exit code.

### Test strategy

- Unit test for `upgradeConfigSchema` (`mkdtempSync`, git-init like
  `loadConfig.test.ts`): with no `phax.json` → `no_config`; with a `phax.json`
  present and no schema → `updated`; immediately again → `current`; after
  mutating the schema file → `updated`. Include a case where `phax.json` is
  invalid JSON/content and assert the schema is still written (`updated`).
- CLI smoke test (`tests/unit/cli/schemaUpgrade.test.ts`) with a fake
  `OutputPort` and `process.chdir`: asserts the three exit codes and printed
  lines.

### Implementation order

1. Write the `upgradeConfigSchema` unit test and the CLI smoke test.
2. Export `locatePhaxConfig` from `src/app/loadConfig.ts`.
3. Add `upgradeConfigSchema` to `src/app/initProject.ts`.
4. Implement `src/cli/commands/schema.ts` and register it in `src/cli/main.ts`.
5. Update `README.md`.

### Excluded scope

- Any modification of `phax.json` content.
- Other `phax schema` subcommands (e.g. `show`, `path`).
- Config-error reporting changes (phase-05).

### Verification

- The `full` gate profile configured in `phax.json`.

### Expected handoff content

- The `phax schema upgrade` surface, the `UpgradeResult` union, and the
  `locatePhaxConfig` export path/signature.
- Confirmation the command never writes `phax.json`.
- Any deviation from the planned file lists, with the reason.

### Commit subject

feat(cli): add phax schema upgrade command

### Commit body

Add `phax schema upgrade`, which regenerates the project's phax.schema.json from
the installed binary's config contract via the shared writeConfigSchemaFile. It
locates phax.json (new locatePhaxConfig export) and writes the sibling schema
only, never touching phax.json, and reports updated vs already-current. Covered by
unit and CLI tests including the invalid-config and idempotency cases.

## phase-05 — Consistent config-validation error reporting {#phase-05-config-error-ux}

**Recommended model:** claude-sonnet-4-6
**Recommended effort:** medium

Make runtime config-validation failures explain themselves and propose a fix, so
the binary's validation (the authority) gives users an actionable, blocking
message in the workflows that matter.

### Detailed instructions

- Add a shared helper `reportConfigError(err: ConfigValidationError, out:
  OutputPort): void` in a new `src/cli/commands/reportConfigError.ts` that prints:
  - the field-level message (`err.message`, which already carries
    `formatParseError` detail) and `err.path` when present;
  - a fix hint: `Fix the reported field(s) in phax.json, then run \`phax
    validate\` to recheck. If your editor isn't flagging this, run \`phax schema
    upgrade\` to refresh the local schema.`
- Wire `reportConfigError` into the blocking workflows that load config and
  currently print an ad-hoc line: `src/cli/commands/run.ts`,
  `src/cli/commands/resume.ts`, `src/cli/commands/resetPhase.ts`. Replace their
  existing `out.error("Config error: …")` branches with `reportConfigError`,
  keeping the same non-zero exit behavior.
- Do not change `loadConfig` or `decodePhaxConfig`.
