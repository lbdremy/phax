# PHAX Spec — Push Branch and Create Pull Request

## Status

This document specifies the final publication step for a PHAX run.

The goal is to let PHAX optionally push the final run branch to a remote and create a pull request.

This is a functional specification.

---

# 1. Goal

At the end of a successful PHAX run, PHAX should optionally:

```txt
push the final branch to the remote origin
create a pull request
include the final review handoff in the pull request description
record the pull request URL in the final instructions
log the pull request URL in verbose mode
```

The first supported provider is:

```txt
GitHub
```

The implementation should use:

```txt
gh
```

The user is responsible for having the GitHub CLI installed and authenticated.

---

# 2. Product principle

PHAX should not require users to manually copy the final review handoff into a pull request.

If the run has completed and produced a review-ready branch, PHAX should be able to publish it as a PR with the relevant review context already attached.

The PR should become the natural external review entrypoint.

---

# 3. Configuration

This feature should be enabled through PHAX configuration.

Default behavior:

```txt
disabled
```

The user must explicitly configure PR publication.

Conceptual configuration:

```json
{
  "publish": {
    "enabled": true,
    "remote": "origin",
    "provider": "github",
    "pushBranch": true,
    "createPullRequest": true
  }
}
```

Provider support for this iteration:

```txt
github
```

Other providers are out of scope.

---

# 4. Required conditions

PHAX should only attempt PR publication when:

```txt
the run completed successfully
the final branch exists
the final branch has committed changes
the repository has a configured remote
the configured remote is origin unless overridden
the review handoff exists
the GitHub CLI is available
the GitHub CLI is authenticated
```

If these conditions are not met, PHAX should fail the publication step with a clear diagnostic.

PHAX should not pretend the PR was created if it was not.

---

# 5. Timing

PR publication should happen after the final review handoff is generated.

Recommended final flow:

```txt
final phase completes
final gates pass
final phase end-of artifact is generated
final file reconciliation is generated
global file reconciliation is generated
review-handoff.md is generated
final branch is pushed to origin
pull request is created
PR URL is recorded
final instructions are updated
run enters review-ready state
```

The PR body should include the final review handoff.

---

# 6. Branch push behavior

When configured, PHAX should push the run branch to the configured remote.

Default remote:

```txt
origin
```

The pushed branch should be the final PHAX run branch.

PHAX should not push unrelated branches.

PHAX should not push if the working tree is in an unsafe or ambiguous state.

If the branch has already been pushed, PHAX should handle that gracefully.

The publication step should be safe to retry.

---

# 7. Pull request creation

When configured, PHAX should create a pull request from the pushed run branch.

Provider:

```txt
github
```

Tool:

```txt
gh
```

The PR should target the repository’s default base branch unless configured otherwise.

Optional config:

```json
{
  "publish": {
    "baseBranch": "main"
  }
}
```

If no base branch is configured, PHAX should use the repository/provider default.

---

# 8. Pull request title

PHAX should generate a PR title from the run.

The title should be deterministic and understandable.

Possible title sources, in priority order:

```txt
explicit configured PR title
run title
plan title
short run name
first phase / plan summary
```

Example:

```txt
PHAX: implement provider-native security mode
```

PHAX should avoid meaningless titles such as:

```txt
Update files
```

---

# 9. Pull request body

The PR body should include the full final review handoff.

Primary source:

```txt
review-handoff.md
```

If the current implementation uses another final aggregated review document name, PHAX should use that existing artifact rather than requiring a rename.

The body should make clear that it was generated by PHAX.

Recommended structure:

```md
# PHAX Run Review Handoff

Generated by PHAX.

<contents of review-handoff.md>
```

The PR body should not be generated by an LLM.

It should be deterministic.

---

# 10. Handling large review handoffs

If the final review handoff is too large for a GitHub pull request body, PHAX should not silently truncate it without saying so.

Acceptable behavior:

```txt
create a shorter PR body with a clear note
reference the review-handoff.md artifact in the branch
include the most important summary sections
report that the full handoff was too large for the PR body
```

The exact size handling can be refined later.

Functional requirement:

```txt
PHAX must not silently lose review context.
```

---

# 11. Idempotency

The publication step should be safe to retry.

If the branch was already pushed:

```txt
do not fail solely because the remote branch already exists
```

If a PR already exists for the branch:

```txt
reuse or report the existing PR
do not create duplicate PRs
record the existing PR URL
```

If the PR was created but final instructions were not updated:

```txt
PHAX should be able to rerun the publication/finalization step and record the URL
```

---

# 12. Final instructions update

PHAX already produces a final instructions artifact for the user at the end of a run.

This feature should add the PR link to that artifact.

The spec does not require renaming the existing instructions file.

The final instructions should include:

```txt
pull request URL
remote branch
base branch when known
publication status
next recommended user action
```

Example section:

```md
## Pull request

A pull request was created for this run:

<PR URL>

Remote branch:

`origin/<branch-name>`

Next step:

Open the pull request and review it using the generated review handoff.
```

If PR creation fails, the instructions should include the failure reason and a suggested manual fallback.

---

# 13. Verbose output

When PHAX runs in verbose mode, it should log:

```txt
publication enabled
remote selected
branch selected
whether branch push succeeded
whether PR creation was attempted
PR URL when available
failure reason when unavailable
```

Example:

```txt
Publishing branch to origin...
Branch pushed: origin/phax/provider-security-mode
Creating GitHub pull request...
Pull request created: https://github.com/org/repo/pull/123
```

The PR URL should be easy to copy from terminal output.

---

# 14. Run metadata

PHAX should store publication metadata in the run state.

Required metadata:

```txt
publish enabled
provider
remote
branch
base branch when known
push status
pull request status
pull request URL
created at
failure reason if any
```

This metadata should be included in:

```txt
run status
verbose logs
trace logs
final report
final instructions
```

---

# 15. Command support

PHAX should support automatic publication at the end of the run when configured.

It should also support a manual retry command.

Recommended command:

```bash
phax publish-pr <run-name>
```

Purpose:

```txt
push the run branch and create or recover the PR publication step
```

This command should be idempotent.

It should be useful when:

```txt
gh was not installed during the original run
gh was not authenticated
network failed
PR creation failed
final instructions were not updated
```

---

# 16. Failure behavior

PR publication failure should not erase or invalidate the completed run.

If the run itself completed successfully but PR creation failed, PHAX should keep all run artifacts and clearly mark publication as failed.

Possible final status:

```txt
review_ready
publish_failed
```

or:

```txt
review_ready_without_pr
```

The exact status name is implementation-specific.

Functional requirement:

```txt
The user must be able to understand that the code run completed but PR publication failed.
```

PHAX should provide enough information to retry.

---

# 17. GitHub CLI requirements

For GitHub publication, PHAX should require:

```txt
gh installed
gh authenticated
current repository recognized by gh
sufficient permission to push branch
sufficient permission to create pull request
```

If `gh` is missing or unauthenticated, PHAX should provide a clear error.

Example:

```txt
GitHub PR publication is enabled, but the GitHub CLI is not available or not authenticated.

Install and authenticate gh, then rerun:

phax publish-pr <run-name>
```

PHAX should not manage GitHub authentication itself.

---

# 18. Security considerations

This feature launches external commands:

```txt
git
gh
```

Therefore it must respect PHAX’s command execution safety model.

PHAX should only run expected publication commands.

PHAX should not concatenate untrusted user input into shell strings.

PHAX should record publication commands in logs/traces.

If PHAX uses a runtime command allowlist, `git` and `gh` must be allowed for this feature.

---

# 19. Out of scope

The following are out of scope for this iteration:

```txt
GitLab merge request support
Bitbucket pull request support
custom provider APIs
direct GitHub API integration without gh
automatic reviewer assignment
automatic labels
automatic milestones
automatic draft/review-ready transitions
automatic merge
automatic CI monitoring
LLM-generated PR descriptions
```

These may be added later.

---

# 20. Acceptance criteria

This feature is complete when:

1. PR publication is disabled by default.
2. Users can enable publication through PHAX configuration.
3. The first supported provider is GitHub.
4. PHAX uses the GitHub CLI for GitHub publication.
5. PHAX verifies that required final artifacts exist before publication.
6. PHAX pushes the final run branch to the configured remote.
7. The default remote is `origin`.
8. PHAX creates a PR from the pushed branch.
9. PHAX uses the repository default base branch unless configured otherwise.
10. PHAX generates a sensible PR title from the run.
11. PHAX includes the final review handoff in the PR body.
12. PHAX does not require renaming the existing final review handoff artifact.
13. PHAX records the PR URL in run metadata.
14. PHAX adds the PR URL to the final instructions artifact.
15. PHAX logs the PR URL in verbose mode.
16. PHAX publication is idempotent.
17. PHAX does not create duplicate PRs for the same branch.
18. PHAX provides a retry command such as `phax publish-pr <run-name>`.
19. If publication fails, PHAX keeps the completed run artifacts.
20. If publication fails, PHAX clearly reports that the run completed but PR publication failed.
21. PHAX gives the user enough information to fix `gh` setup and retry.
22. This feature does not implement non-GitHub providers.

---

# 21. Product summary

PHAX should be able to finish a run by opening the review path automatically.

When configured, the final outcome should be:

```txt
code committed
branch pushed
pull request created
review handoff included in PR body
PR link written into final instructions
PR link visible in verbose output
```

The core rule is:

```txt
A completed PHAX run should be able to become a reviewable pull request without manual copy-paste.
```
